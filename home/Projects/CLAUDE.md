# ~/Projects 共通の指示（配下の全プロジェクトに適用）

## PDF抽出に着手する前に必ず決めること：見開き／横並びページの扱い

PDF からの抽出・ページ画像化に取りかかる前に、**まずユーザーに「見開き／横並びページをどう扱うか」を確認し、方針を確定してから進める**こと。後から方針を変えると、全ページの分割・通しページ番号・図の紐付けがやり直しになる。配下プロジェクト（seg-math-texts の数学、PDF の理科・社会など）で実際に出たパターンを踏まえ、**着手時に次から選んでもらう**（複数該当もあり）。

1. **単ページ**：1 PDFページ = 1 実ページ（縦長）。分割しない。
2. **左右見開き = 2実ページ**：1 PDFページに実ページが左右2枚（横長）。→ 左右に分割し各々を1ページとする。
   - **分割方法**を選ぶ：
     - **(a) 機械的に中央 `w//2` で真っ二つ**（中身を一切見ない）。※ **seg-math-texts はこれを絶対ルール**とする。
     - **(b) レイアウト検出で段境界（中央の罫線／ガター）を見つけて分割**（PDFプロジェクト方式。中央線が無い等で効かないページは単段にフォールバック）。
   - **読み順**を宣言：標準は 左→右。
3. **1ページ内の2段組**：1実ページの中に2列（left/right）。→ 列ごとに分割し **左段→右段** の順に読む。OCR も列ごとに行い、`full`（左右結合）から一度に起こさない。
4. **左右で内容が別系統（クロス配置／反転綴じ）**：左ページと右ページで扱う単元・大問が異なる、または問題側と解答側が左右反転している。→ **作業の最初に「左＝○○側／右＝○○側」を明示宣言**し、**左右で別の大問を建てる**（同一大問にまとめない）。
5. **混在・不規則ページ**（見開きが表紙のように見える等）：自動分割に頼らず、**手動クリップ（region_app 等）で人が範囲指定**。

→ どれに当たるかを**着手時に必ずユーザーへ問い、決めてから**ページ画像化・抽出に進む。各プロジェクト固有の分割規則（例：seg は必ず 2-(a) の機械分割）は、そのプロジェクトの CLAUDE.md に従う。

## PDF・画像からの日本語テキスト抽出（共通ワークフロー）

紙面 PDF やスキャン画像からテキストを起こすときは、次の役割分担で行う。**NDL-OCR Lite を「検証用」ではなく「最初に使う一次抽出ツール」として使う。**

1. **日本語の本文抽出は、まず NDL-OCR Lite を使う（第一選択・一次抽出）。**
   - 実行: `ndlocr-lite <画像ファイル|画像ディレクトリ> <出力ディレクトリ>`
   - 出力: 出力dir に `<stem>.txt`（認識テキスト）/ `.json` / `.xml`。**CPUで約1秒/枚・GPU不要**。
   - 実体: `~/tools/ndlocr-lite`（専用 uv venv 同梱）。国立国会図書館（NDL）公式の ONNX 軽量OCR。
   - Python CLI（`src/ocr.py`）をラッパー越しに呼ぶ。手動用のデスクトップGUI版も別途あるが、自動化では CLI を使う。

2. **数式・記号・図・レイアウト依存の箇所は AI（vision）で起こす／検証する。**
   - NDL-OCR は `$x^2$`・`±`・分数・図・複雑なレイアウトを苦手とする（`x^2`→`x2`/`r2`、`±`→`土`、`0`→`Q` のように崩れる）。ここは AI が担当・補完する。
   - 逆に、細かい平文の日本語漢字は NDL-OCR が高精度で、AI（vision）の見落としを拾えることがある。**両者は相補的**——平文＝NDL一次、数式・記号＝AI、最終照合＝AI。

3. **コツ:** 2up（見開き2ページ）画像は、印刷1ページ単位に左右分割してから OCR にかけると精度が上がる。ツールのパスは ASCII のみ（全角パスでは起動しないことがある）。

> まとめ: **「平文の日本語＝NDL-OCR が一次抽出、数式・記号・図＝AI が担当、全体を AI で照合・校正」**。

## Python 環境
Python は終始一貫 `uv` を使う（詳細は `~/.claude/CLAUDE.md`）。外部ツール（ndlocr-lite 等）も専用の uv venv で動かす。

### Python の起動コマンドは `uv run python …` で統一（配下全プロジェクト共通）
`~/Projects` は uv ワークスペース（共有 `.venv`＝Python 3.14系。`pyproject.toml`/`uv.lock` は dotfiles 実体のシンボリックリンク）。スクリプトもアプリも **`uv run python <script>.py …`** で起動する。cwd 非依存で workspace の `.venv` を自動解決し、依存も自動同期される。

- **やらない**：素の `python3 xxx.py`（＝システムの **/usr/bin/python3 3.9系** を踏み、依存が無くて落ちる）／素の `python`（PATHに無い）。`.venv/bin/python` でも動くが、`uv run python` に寄せる。
- **docs・スクリプトの起動例も `uv run python` で書く**。シェルスクリプトで変数化するなら `PY=(uv run python)`（bash配列）＋ `"${PY[@]}" script.py`（`"$PY"` だと1語化して壊れる）。
- **対象外（uv を使わない正当な例外。`uv run` に変えない）**：
  - **インストール idiom** `uv venv` / `uv pip install --python .venv/bin/python -r …`（venv 作成・依存導入。`uv run` 前段なので別物）。
  - **Ubuntu 本番サーバ**（systemd の `/usr/bin/python3`・apt 導入の最小 venv 等）。
  - **vendored 外部ツール**（`vendor/ndlocr-lite` 等＝専用 venv）。
  - shebang（明示起動するので実害なし）。
- **新しい Python リポジトリ**は uv プロジェクト化（自前 `pyproject.toml`）か workspace member に追加してから `uv run` を使う（非プロジェクトのフォルダでは共有 venv を解決できない）。
